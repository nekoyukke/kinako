
📝 Kinako Programming Language Specification (Draft)
Kinako は、「Rubyのような美しさとオブジェクト指向の魂」を持ちながら、「C/C++のような堅牢さと低層への制御」を両立し、未定義動作（UB）と曖昧さを徹底的に排除することを目的とした次世代システム言語です。

💎 1. コア・フィロソフィー（哲学）
Kinakoの設計は、以下の3つの絶対的な指針に基づいています。

Deterministic Elegance（決定論的な優雅さ）

世界に根付いている物理的な限界（ゼロ除算など）以外の**「人工的なUB」を完全に抹消**する。

実行時の振る舞いは、コードから100%予測可能でなければならない。

Explicit Intent（意図の明示）

「暗黙的な型変換」を禁止し、すべての変換はプログラマの意思による「明示的な記述」を求める。

Absolute Clarity（絶対的な明瞭さ）

「どちらの関数が呼ばれるか分からない」「状態が不明」といった「曖昧さ」を排除する。

🏗️ 2. オブジェクト指向設計（継承の廃止とメッセージング）
Kinakoは、データ構造を共有する「継承」を廃止し、オブジェクト同士がメッセージを送り合う「ネットワーク」としてのOOを採用します。

継承の廃止: データの密結合と曖昧なオーバーライドを避けるため、クラス継承は存在しない。

インターフェース主体: 「何であるか」ではなく「何ができるか（契約）」を重視する。

委譲と譲渡: 機能の再利用は、明示的な「委譲（Delegation）」や「所有権の譲渡（Move）」によって行う。

🛡️ 3. インターフェースと契約（The Interface Protocol）
Kinakoのインターフェースは、単なるメソッドリストではなく、**「安全な通信のための厳格なプロトコル」**です。

構成要素
契約 (Contract): メソッド名、引数の型、戻り値の型の定義。

事前条件 (where句): メソッドが実行可能な条件（ガード）の定義。

違反時の挙動 (else句): 条件を満たさない場合の振る舞い（パニック、規定値、等）の明示。

インターフェース定義例
C++
interface Stack<T> {
    fn T pop(borrow self) 
        where {
            self.len > 0 => this(),       // 成功時はクラス側の実装(body)へ
            else => panic("Stack empty")  // 失敗時は安全に停止
        };
}
💻 4. クラスと実装（Class & Body）
クラスはインターフェースの契約に従い、**「保証された安全な状態」**でのみロジックを記述します。

クラス実装例
C++
class MyStack<T> implements Stack<T> {
    let list<T> data = [];
    let int len = 0;

    // インターフェースの where 句を通過した後にのみ実行される
    body T pop(borrow self) {
        let T val = data.remove_last();
        self.len -= 1;
        return val;
    }
}
⚙️ 5. 安全性と型システム
None-Safety: string や int などのリテラルはデフォルトで null/none を受け付けない。

Ownership & Borrowing: Borrow (借用), Mut (可変借用), Move (所有権譲渡) を文法レベルでサポートし、メモリ管理のUBを防ぐ。

静的証明: 可能な限り、where 句の条件が満たされていることをコンパイル時にチェックする。
