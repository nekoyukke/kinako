# Philosophy

## なぜ Kinako を作るのか

既存のプログラミング言語には、それぞれ優れた特徴がある。

Kinakoは、それらの機能を否定するための言語ではない。\
むしろ、それらが解決しようとしている問題を認めた上で、より少ない概念と、より単純な規則によって実現できないかを探求することを目的としている。\
そのためKinakoでは、「機能を増やすこと」よりも「概念を減らすこと」を設計上の重要な価値としている。

### C++

まず、最も有名な言語`C++`だ。\
`C++`では高い抽象化能力をもち、様々なプログラミングスタイルを提供することに成功している\
一方で、それらは抽象化・汎用性・後方互換性を実現するために、多くの概念や規則を導入している。\
資源管理の方法が型や修飾子として多数存在する。
`&T`, `std::weak_ptr`, `std::unique_ptr`, `std::shared_ptr`などに加え、`const T`などと複雑なメモリ管理システムが存在する。

### Rust

次に`Rust`だ。\
`Rust`はさまざまな抽象的なクラス設計を取り入れているほか、`所有権`というシステムで明確に値の生存時間を設定している。\
事実、とても人気で 愛される言語に 9年連続で一位 を取り続けている。\
一方、`所有権`は所有権規則を満たすために設計変更が必要となる場面がある。\
さらにデフォルトの添字アクセスではPanicが起こるなど、安全とは言えない場合もある。

## 解決したい問題

### 責務の混在

`C++`では`&T`, `std::weak_ptr`, `std::unique_ptr`, `std::shared_ptr`, `const T`\
というとても複雑な権利システムが存在する。\
例えば`std::shared_ptr`は`RAII`や`所有権`, `型`などの責務が混在してる。
また、`Rust`では所有権チェッカにより問題は解決しているように思えるが、\
`Arc<T>`, `Mutex<T>`, `Box<T>`, `Rc<T>`, `Cell<T>`などと、複雑なマルチスレッド用のシステムが存在する。

### 不意なPanic, エラー

`Rust`ではデフォルトの添字アクセスではPanicや\
このようなことは通常の制御フローにおいて終了を伴う失敗が発生する場合がある。

### 暗黙的な挙動

`Rust`では代入演算子により、`move`が自動的に実行される。\
また、`Python`においては`@dataclass(frozen=True)`やプリミティブ型以外では参照的コピーが行われる。\
このような挙動はユーザーが実際の値の所有状況の挙動を想像することができなくなり、\
実装元を確認しなければならないことや、例えば`Pin<T>`などの特殊な型での挙動が一部不安定になってしまうことがある。

### 実用性

`Rust`は特定のデータ構造やアルゴリズムでは設計上の制約を受ける。\
また、所有権チェッカによるコンパイル時間の増加傾向がある。\
このような状態はあまり実用的ではない。

## 設計原則

Kinakoではこれらの問題を、次の設計方針によって解決することを目指す。

### 一つの概念は一つの責務のみを持つ。

一つの概念が複数の責務を持つと、仕様の理解・実装・拡張の全てが難しくなる。\
Kinakoでは概念ごとの責務を明確に分離し、それぞれを独立して設計できるようにする。

 * `Type`は値のみを示す
 * `Right`は静的な値への管理方法のみを示す
 * `policy`は動的な実装戦略を示す

### 新しい問題に対して、新しい概念を増やすことを第一の解決策としない。

新しい問題が現れても、まず既存概念の組み合わせで表現できないかを検討する。概念を増やすことは最後の手段とする。

### 同じ規則は、言語全体で同じ意味を持つ。

同じ構文や規則は、文脈によって意味が変化しないことを目指す。\
特殊ケースよりも一般的な挙動、一貫性を優先する。

### 可読性を優先する

一時的な記述量の削減よりも、コードを読む人が理解しやすいことを優先する。\
そのため、可読性を損なうだけの糖衣構文や新しい概念は導入しない。

### 明示的な動作を可能な限り優先する。

プログラムの重要な意味が暗黙に変化することを避ける。

### 回復可能な失敗は型で表現する。

回復可能な失敗は通常の制御フローの一部であり、例外やpanicではなく型によって表現する。

### 実行時よりコンパイル時に検出できるものはコンパイル時に検出する。

実行前に検出できる問題は可能な限りコンパイル時に報告する。

### 同じ意味なら、実装戦略は交換可能である。

同じ意味を持つなら、実装方式は交換可能であるべきである。利用者は意味を指定し、実装方式はPolicyによって選択できる。

## トレードオフ

 * `kinako`では記述量が増えても、設計原則を優先する
 * 暗黙な最適化よりも明示的な最適化を実装する
 * 学習コストよりも設計原則を優先する。
 * 利便性よりも一貫性を優先する。

## 今後の方針

新しい機能はこの文書と矛盾しない場合のみ採用する。
仕様は変更される可能性があるが、その変更は本設計原則を維持・強化する方向でのみ行う。
