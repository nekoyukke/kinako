# kinako language

## 思想

この章は読み飛ばしてもらっても構いません。

- `C言語`は安全ではない
- `C++`は`virtual`やテンプレートなどで複雑怪奇
- `Rust`は素晴らしいが複雑であることがある
- `Python`は便利だが配布が困難
- `JavaScript`は`DOM`などで便利だが型やクラスなどが危険

このように、これらの既存言語には「スピード」「手軽」「安全」などの強みがあるが「安全性」「複雑性」「型安全性」「学習コスト」などの決定的な弱点を抱えている\
そこで私はこれらの問題に不満があるため回避できるような言語を作成するにあたった。\
では、どのように回避するのか。

既存言語に存在する弱点を避けるために`kinako language`では、まず「何」を優先して「何」を捨てるかを明確にする必要性がある。\
`kinako language`では、次のような思想を採用した。

### 暗黙的の徹底排除

暗黙的で済まされてきたUB（未定義動作）やバグなどは多い。
例えば
```:c
int i = 42;
int a = i++ + i++;
```
の様なコード。\
このようなコードを排除するために従来の言語では暗黙的だったものをすべて明示的に変える方針を採用した。\
次のような仕様である
- 型推論は極力最低限に。
- [この章](#単純なルール)でも語られているが一般に演算子の前後で型は変わらず、左右で常に同じ型になる。
- 型変換は明示的である。nに依存するオーダーであればメソッドを使用するなど

### 単純なルール

従来の言語、例えば `Python` では`\`演算では`float`などを返し、`*`演算では演算内容にもよるが`str`や`int`, `float`などを返す。\
このような例外だらけの言語仕様はあまり美しくなく、統一性に欠ける。\
私の言語では基本的に演算は`式`, `単純演算`, `比較`, `特殊演算`に分ける。

`式` では文脈により、意味が変わるもの。例えば変数や関数実行等。\
`単純演算` では文脈によらず、演算子の前後で型は変わらない。また、演算子の左右は常に同じ型であるべきである。\
`比較` では常に`Boolean`値を返す。\
`特殊演算` では演算子の左右で違う値などを返す。唯一の例外のある仕様である。

### 占有権

ここがこの言語でのもっともな目玉であると確信している。

従来の言語では型が複雑である。\
例えば`C++`の`std::unique_ptr`や`std::shared_ptr`など。\
半端な所有権を持たせたことにより複雑怪奇になっている要因であろう。\
また`Rust`はきちんとした所有権を定義しているが一部複雑な形になっている。

ここで私が提唱するのが `占有権(Possession)` システムだ。\
従来の言語では `型(Type)` に依存しすぎてしまっている。ここでいい例が `C++` のスマートポインタだ。\
本来ならばコピー/代入ができた部分で `std::unique_ptr` であればできない。\
[これは例外だらけで美しくない。](#単純なルール)\
また`Rust`は一見、`所有権`という部分でこの思想に合致するように思えるが、`Rc<T>`や`Pin<T>`等により型が増えてゆき結局のところ`c++`よりも煩雑になってしまったと感じている。

`型(Type)` はあくまでオブジェクトの構造、動作を定義するものであり、\
変数たちのライフタイムの開始点や終了点、破棄といった責務まで抱えるべきではない。\
それら（変数の再代入やコピーなど）は `占有権(Possession)` によって責務を明確に管理されるべきである。\
`型(Type)`から`占有権(Possession)`を切り離すことで、`C++`の複雑怪奇な型や`Rust`の所有権に起因する複雑な型の増殖を回避できると確信している。

`占有権(Possession)`はいくつくかのフラグのオンオフであり、単純な演算で合否がわかる。

勘のいい読者は「あれれ～？おっかしいぞぉ～？`List`など可変長コンテナーオブジェクトなどは本来動的だから占有権として定義すべきなのにその項がないぞぉ～」と思っただろう。\
私は勘のいい読者は ~~嫌いだ~~ 大好き。\
この一見不可解な構造は次のような仕組みを採用することで成り立っている。

#### 占有権の下方適用
変数に適用された占有権は下方適用する。\
例えば`List<List<int>>`のような型でも変更不可能なら内部も変更不可能。

#### 占有権の一部例外
ただし、どうしても下方適用から外したい！という例から、一部例外を採用する。\
明示的に例外であることを表せば、安全に取り扱える。ただしアンカーに汚染される。

#### 占有権のアンカー
アンカーは占有権が正しく下方適用されているかを許容する。\
もしついていればそれはアンカーに汚染されている。アンカーに汚染されれば、\
特殊な操作や一部の操作が例外的に適用できない場合もある。\
また、共有が難しい占有権同士だった場合、宣言すること自体が罪になる。

わからない人へ:\
例えば君が弱い制約の占有をしたバッグを持っていたとしよう。\
バッグはあなただけが持てる。(`unique`)だけど、\
もし、君のバッグの中に図書館から借りたもの（`shared`）が紛れ込んだらどうしようか。\
...困るよね。だって`unique`と`shared`は同じにできない。\
じゃあどうするか。\
...そうだね。アンカーを使おう。そうすれば図書館から借りたものも返せるし（その間はバッグをいじれないけどね）\
君の大切なクレヨンが見つかるかもしれない。


### インターフェイス

従来の言語ではオブジェクト指向は`カプセル化`・`メソッド`・`継承`として紹介されているが、\
オブジェクト指向のアンチパターンの多くは継承が由来となっていることが多い。\
私の言語ではオブジェクト指向を`カプセル化`・`メソッド`のみにし、継承に準ずるものを`interface`として提供をする。\
`interface`は`class`で実装すべきものを集約したメソッド群であり、\
`class`にセットすることにより、`interface`メソッドが自動的に定義されなければならない。

## 構文(Syntax)

### 変数宣言
変数宣言では占有権と型が必要である。
```
<占有権> <型> <識別子>;
または
<占有権> <型> <識別子> = <右辺>;
```

### 式
式は演算を持つ。
```
<演算>;
```

### 引き入れ文
指定のファイルを読み込む
```
import <パス>;
```

### 分岐文
分岐をする
```
if <条件> <文>
または
if <条件> <文> else <文>
```

### 繰り返し文
繰り返す
```
for <変数名> in <リストオブジェクトまたはイテレーター（未実装）> <文>
```

### 条件繰り返し文
繰り返す
```
while <条件> <文>
```

### 帰り文
カエル
```
return <式>;
```

### 関数宣言
関数を宣言する
```
fn <型> <識別子>(<引数>) <文>
<引数>は
<型> <識別子>,の組
```
