- `C言語`は安全ではない
多くの場合、「ポインタ渡し」などによる意外な副作用。\
高級言語だと思ったら、読みやすいただのアセンブリのような感覚で書く羽目になる。
- `C++`は`virtual`やテンプレートなどで複雑怪奇
C++は高い抽象化能力と性能を両立している。\
しかし継承、virtual、template、特殊メンバ関数など\
複数の抽象化機構が相互作用することで\
言語全体の理解が難しくなっている。
- `Rust`は素晴らしいが複雑であることがある
安全性はRustの所有権・ライフタイムなどの個々が難しいものによって構成される。
- `Python`は便利だが配布が困難
インタープリタ言語のため、遅く、exe配布などが困難である。
特に`Cython`などを使えばよいかもしれないが、リファクタリングが必要な場合もあり、なかなか導入が進まない。
- `JavaScript`は`DOM`などで便利だが型やクラスなどが危険
特にtupleで便利に、ユニークにあらわせるが、そこが仇。
TSを使えばよいが、JSのコードも実行できるため、そこまで。

このように、これらの既存言語には「スピード」「手軽」「安全」などの強みがあるが「安全性」「複雑性」「型安全性」「学習コスト」などの決定的な弱点を抱えている\
そこで私はこれらの問題に不満があるため回避できるような言語を作成するにあたった。\
では、どのように回避するのか。

既存言語に存在する弱点を避けるために`kinako language`では、まず「何」を優先して「何」を捨てるかを明確にする必要性がある。\
`kinako language`では、次のような思想を採用した。

### 権利

多くの言語では型に多くの性質を持たせてしまっている。\
例えば`C++`の`std::unique_ptr`は値の型であると同時に、資源管理戦略も表現している。\
これにおいて資源管理で優秀とされる`Rust`でも`Arc<T>`や`Rc<T>`などで、同類である。

しかし、このように資源管理責務を型に担わせることで、理解・拡張性が大幅に低下する。\
そこで、私の言語では、型と権利を分離する。\

型は値の構造を表現する。\
権利は変数が値をどのように扱えるかを表現する。

権利はいくつかの軸から構成され、それらのタプルで構成される。\
例:

- Access
    - Read
    - Write

- Identity
    - Shared
    - Unique

ある軸に置いて下位の者をそれより上位の者は内包する。\
つまり、Accessという軸に置いて、\
Write => Readという上下関係が成り立つ。\
するとWriteはReadを内包する。\
型と権利の分離により、アクセス制御のルールが型定義に混入しなくなる。

従来の言語では、所有・共有・借用などが独立した概念として扱われるが、\
kinakoではこれらを権利軸の組み合わせとして統一する。\
`kinako`では`Onwer`ですら、簡単ないくつかの軸同士のタプル、\
例えば`Unique + Write`のような構造で表せられる。\
これは多くの型に悩まされてきた人にとって理解がしやすく、\
手につきやすいくわかりやすいものを提供することができる。

これにより、任意な権利を明確な共有方法、明確な管理方法で表すことができる。

### ポリシー

ポリシーは権利だけでは表せられない、実行・運用の制約である。\
例えばrustの`Arc`や`Mutex`が該当する。\
ポリシーは権利と違って実装に置いての実行戦略および運用制約であり、
ほとんどの場合において型と同軸に置かれていた資源管理・同期・共有方法などを分離する。\
ユーザ側では方向性を定め、それによりその実装が担保される実装に落とし込まれる。


### インターフェイス

`interface`は`class`が実装しなければいけないメソッド集合の実装契約である。
`class`に適用された場合、`interface`のメソッド群が実装されることを保証する。

疑似コード
```
interface Speak {
    fn void speak() ...
}

Group Animal use Speak;

class Dog use Speak in Animal {
    fn void speak() {
        print("one!");
        return;
    }
}

class Cat use Speak in Animal {
    fn void speak() {
        print("two!");
        return;
    }
}


list[Animal] zoo = {
    Cat(),
    Dog(), ...
};

for animal in zoo {
    animal.speak();
}
// one!
// two!
// ...
```

`Group`は型システムではなく、
値の分類を行うための実行時ラベルであり、\
継承ではなく値を集合として扱うための構造である。\
構文から参照され、制御が発生する場合がある。

また、この仕様により、自動的にコードが
`interface`, `Group`, `class`のような順の構造になる。
